## Why SSH, Not kubectl exec

The fo_scheduler dispatches scan agents to remote hosts over SSH. Three
reasons why we kept this rather than switching to `kubectl exec`:

1. **Security**: SSH dispatch requires only a `Role` with `get/list/watch`
   on pods — the minimum needed for service discovery. `kubectl exec`
   would require `create` on `pods/exec`, a much broader permission that
   security teams routinely flag.

2. **Compatibility**: The existing scheduler engine (`agent_spawn()` in
   `agent.c`) already has a mature SSH code path. Rewriting it to use
   the Kubernetes exec API would mean reimplementing process lifecycle
   management, stdin/stdout piping, and signal handling — all of which
   SSH handles natively.

3. **Reliability**: SSH is battle-tested for exactly this use case (remote
   process execution). The Kubernetes exec API has known issues with
   connection drops, WebSocket upgrades through proxies, and inconsistent
   timeout behavior that would require extensive workaround code.

## Dynamic [HOSTS] Registration

### How It Works

The scheduler reads its host list from the `[HOSTS]` section of
`fossology.conf`. In a Kubernetes deployment, this section is dynamically
generated by `render_fossology_conf.py`:

```
[HOSTS]
fossology-workers-0 = fossology-workers-0.fossology-workers-headless.fossology.svc.cluster.local /usr/local/etc/fossology 2
fossology-workers-1 = fossology-workers-1.fossology-workers-headless.fossology.svc.cluster.local /usr/local/etc/fossology 2
```

### Init Container vs Sidecar

| Phase | Container | Mode | Purpose |
|-------|-----------|------|---------|
| Startup | `render-config` (init) | `--mode=once` | Blocks until `minReadyStartup` workers are ready, then writes initial config |
| Runtime | `config-sync` (sidecar) | `--mode=loop` | Continuously watches for pod changes, rewrites config, triggers `fo_cli --reload` (with `SIGHUP` fallback) |

### What Happens When a Worker Pod is Deleted

1. Kubernetes removes the pod from the headless service endpoints
2. config-sync detects the change on next poll cycle (`pollIntervalSeconds`)
3. config-sync renders a new `fossology.conf` without the deleted worker
4. config-sync runs `fo_cli --reload` against the in-pod scheduler and falls back to `SIGHUP` if needed
5. `fo_scheduler` reloads configuration via `scheduler_config_event()`
6. Any agents running on the deleted worker are marked as failed
7. Their jobs are re-queued to remaining workers

## Per-Agent-Type Scaling (PR3 Feature)

### How agents= Maps to Dedicated Worker Pools

With the PR3 extension, each host in `[HOSTS]` can specify which agents
it accepts:

```ini
[HOSTS]
workers-nomos-0 = workers-nomos-0.fossology-workers-headless.ns.svc.cluster.local /usr/local/etc/fossology 5 agents=nomos,ojo,monk
workers-copyright-0 = workers-copyright-0.fossology-workers-headless.ns.svc.cluster.local /usr/local/etc/fossology 5 agents=copyright,ecc,keyword,ipra
workers-generic-0 = workers-generic-0.fossology-workers-headless.ns.svc.cluster.local /usr/local/etc/fossology 3
```

- `workers-nomos-0` only receives nomos, ojo, and monk jobs
- `workers-copyright-0` only receives copyright-family jobs
- `workers-generic-0` has no `agents=` → accepts all job types

The Helm chart exposes the same model through `workers.pools`. Each pool
creates its own StatefulSet and marks pods with:

- annotation `fossology.org/agent-caps=<comma-separated agents>`
- label `fossology.org/max-agents=<capacity advertised in [HOSTS]>`

The scheduler startup validation path also honors these capabilities now:
hosts that do not advertise a given agent are skipped during the initial
remote agent test instead of being marked failed for an unsupported binary.

### How KEDA ScaledObjects Trigger Per Pool

```
jobqueue (pending nomos jobs)
    │
    ▼
KEDA ScaledObject (nomos)
    │ targetQueryValue: 3
    ▼
fossology-workers-nomos StatefulSet
    │ scales 1→10
    ▼
config-sync detects new pods
    │ renders agents=nomos,ojo,monk
    ▼
fo_scheduler routes nomos jobs → nomos pool only
```

## What Was Deliberately Not Changed

1. **fossology.conf format** — Extended with optional `agents=` field,
   not replaced. Existing configs without `agents=` work identically.

2. **SSH dispatch mechanism** — The core `agent_spawn()` SSH path in
   `agent.c` is untouched (except the PR1 bug fix). No new transport.

3. **Agent binary structure** — Agents are still installed at
   `AGENT_BINARY` (`%s/%s/%s/agent/%s`) paths. Worker pods use the
   same FOSSology image with all agents pre-installed.

4. **Why**: Two previous GSoC attempts that tried to replace the SSH
   dispatch with Kubernetes-native mechanisms (kubectl exec, gRPC) never
   merged because they required changes across too many subsystems.
   This integration succeeds by minimizing the blast radius.

## Known Limitations

1. **Scheduler is single-replica** — `fo_scheduler` maintains in-memory
   state (job queues, agent tracking) that cannot be shared across
   instances. This is a FOSSology architectural limitation, not a
   Kubernetes limitation. The scheduler is not the bottleneck — workers
   are.

2. **ReadWriteMany storage required** — The repository PVC must be
   writable by multiple pods simultaneously. Not all cloud providers
   offer cheap RWX storage. See the [production guide](production.md)
   for options.

3. **KEDA operator requires separate installation** — the Helm chart can
   now render `ScaledObject` resources, but the KEDA operator still must be
   installed cluster-wide before those resources can do anything.

4. **Config-sync poll interval** — Worker pod changes are detected by
   polling (default 5s), not streaming. There is a brief window after
   a pod becomes ready where the scheduler doesn't know about it yet.
